### 1. Meta Brain2Qwerty v2：非侵入式脑机接口逼近开颅水平，代码全开源

* **核心突破**：Meta 发布 Brain2Qwerty v2，基于端到端深度学习，从脑磁图设备记录的原始脑信号中实时解码自然句子，单词准确率达 61%，在最佳受试者上达 78%，而此前非侵入式方法仅约 8%。
* **技术细节**：训练数据来自 9 名志愿者每人约 10 小时的真实打字记录，共 22000 条句子；对大语言模型进行针对神经数据的微调，利用语义上下文将嘈杂脑信号转化为连贯语言。
* **开源生态**：Meta 同时开源了 v1 和 v2 的完整训练代码，合作方 BCBL 同步公开 v1 数据集；Meta 还设立了 500 万美元专项基金支持开放数据集建设。
* **行业意义**：此举为全球数千万因脑损伤无法正常交流的患者提供了非侵入式解决方案，标志着非侵入式脑机接口向临床实用迈出关键一步。

- 每个 skill 文件（共 7 个）对应开发阶段的不同知识，coding agent 按需读取，而非一次性塞入上下文

**技术亮点：**  
基于 ADK（Google Agent Development Kit），提供端到端 Agent 生命周期管理，从本地开发到生产部署。支持的 coding agent 包括 Antigravity CLI、Claude Code、Codex 等，不绑定特定编辑器。评估模块支持故障模式聚类（`eval analyze`）和自动优化（`eval optimize`），可沉淀为可复用的评估链路。

### Qwen 3.6 27B is the sweet spot for local development

970 pts · 636 comments · [quesma.com](https://quesma.com/blog/qwen-36-is-awesome/)

**📌 内容总结**

- **背景/作者意图**: 作者过去对本地模型失望，但认为 Qwen 3.6 27B（密集模型）是首个“在通用智能上有意义”的本地模型，推荐用于本地开发。文章提供了在 MacBook 上通过 llama.cpp 运行的详细指南和性能数据。
- **HN 关注点**:
  - 本地模型的实际可用性——是否真能用于"实际工作"
  - 硬件门槛（RAM/VRAM 需求）与成本
  - 与替代模型（Gemma 4、DeepSeek V4 Flash）的对比

**💬 讨论总结**

- **共识观点**:
  - Qwen 3.6 27B 是当前本地运行模型中性能和可运行性的"甜蜜点"，在合理量化下可运行于 24GB+ VRAM 或 64GB 统一内存的机器。
  - 27B 密集模型质量优于同尺寸 MoE 模型（如 35B A3B），但速度更慢；选择取决于对质量 vs. 速度的取舍。
  - 8-bit 量化（Q8_0）对质量影响不大，是推荐的起点。更激进的量化会显著影响输出质量。
- **工程经验**:
  - **运行配置**:
    - `llama.cpp` 在 Apple Silicon 上实际性能与 MLX 相当甚至更优（需配置 `-fa on` 和 `-ngl 999`）。
    - 多 Token 预测（MTP）可有效提升推理速度（27B 从 18 tok/s 提升至 32 tok/s）。
    - Qwen 3.6 27B 在 Q4_K_M 量化下约需 17GB，可运行于 24GB 显存的卡（如 RTX 4090/5090），速度可达 50 tok/s。
  - **硬件选择**:
    - 128GB MacBook Pro（$6k+）被认为是昂贵的选择。一个常见建议是使用 Mac Mini（64GB + 远程连接）或专用桌面 GPU（如二手 RTX 3090，~$1k），性价比更高。
    - Intel Arc B70（32GB VRAM，~$1200）被提及为另一个可行选项，带宽略低于 M5 Max 但显存充足。
  - **实际使用限制**:
    - 多个用户报告 Qwen 在长上下文或复杂 Agent 任务中可能陷入"思考循环"，需要有效的上下文管理和提示词工程。
    - 该模型在零样本的绿色字段项目上表现良好，但在修改现有、不熟悉的代码库时，效果明显弱于前沿云模型（如 Claude）。
- **反对意见**:
  - 多位用户认为本地模型（包括 Qwen 3.6）对于"实际开发工作"仍然是浪费时间。主要理由是：硬件昂贵、质量不足以替代云 API、在旧代码库上表现差、运行时发热和风扇噪音影响使用体验。
  - 关于作者使用的 128GB 设备，多位用户认为这严重脱离普通开发者的硬件现实，并质疑其成本效益。
- **历史背景**:
  - 多位用户指出，本地可运行模型的进步速度令人印象深刻，如 5 年前的"玩具模型"现在已能胜任部分实用任务。
  - 3090（2018年发布）和 1080（2016年发布）等老 GPU 仍能运行这些模型，说明硬件本身并非主要制约，内存带宽才是。

### LongCat-2.0, a large-scale MoE model with 1.6T total and 48B Active

**📌 内容总结**

- **背景/作者意图**: 美团（Meituan）发布并开源 LongCat-2.0，一个 1.6T 总参数、48B 激活参数的 MoE 模型。技术亮点包括：完全基于 AI ASIC 超算集群训练（非 NVIDIA GPU）、引入 LongCat Sparse Attention 和 N-gram Embedding 架构改进。
- **HN 关注点**:
  - 这是否是对 DeepSeek V4 架构的二次训练/微调？
  - 在非 NVIDIA 硬件上实现大规模训练的实际意义
  - 模型权重的实际可获取性与运行门槛

**💬 讨论总结**

- **共识观点**:
  - 架构设计与 DeepSeek 系列有显著相似性，但作者引入了 LSA（稀疏注意力和索引器优化）和 N-gram Embedding 等实质性创新，并非简单微调。
  - "完全在 AI ASIC 超算集群上训练"是本文最有价值的新闻。多位用户指出，这证明了在没有 NVIDIA GPU 的情况下，仍能进行前沿规模的训练，这对地缘政治和中国算力战略意义重大。有用户猜测使用的是华为昇腾 910C 芯片。
- **工程经验**:
  - **架构细节**: N-gram Embedding 通过 100 倍的嵌入空间扩展来捕获局部上下文，作者认为在 97% 稀疏度的 MoE 下，将参数预算投入 N-gram 嵌入比增加专家数量的收益更高。
  - **训练规模**: 在 5 万+ AI ASIC 上完成，采用 6D 并行（含 EMBP、Superpods 设计）、Muon 优化器、ZeRO-1 和 OOM-aware 卸载。强调需要在高失败频率的非 NVIDIA 硬件上实现确定性计算和容错恢复。
- **风险/限制**:
  - **权重未开放**: 截至发稿时，Hugging Face 仓库和 GitHub 链接均返回 404，权重尚未发布。多位用户对此表示怀疑，并指出该公司过往有"放鸽子"的记录。
  - **运行门槛**: 1.6T 参数意味着即使在 2-bit 量化下也需要约 400GB VRAM，普通硬件基本无法运行。一位用户用涉及钚-241 的核物理知识测试，发现 LongCat-2.0 的回答质量明显低于 Qwen 3.7 Plus 和 Gemini Flash。
  - **基准测试**: 虽然发布了大量基准分数，但用户提醒这些分数大多为"内部测量"，且与 Opus 4.8 等闭源模型仍存在明显差距。
- **商业现实**:
  - 美团（一家外卖/生活服务公司）能训练出此级别模型，被用来类比 Amazon 孵化 AWS 或 Uber 在基础设施上的技术溢出——说明中国互联网巨头在 AI 基础设施上的投入深度。
  - 多位用户认为，这对 Anthropic 和 OpenAI 来说不是好消息，因为开源模型正在迅速逼近，而替代硬件路径正在被验证。

- **HN 关注点**:
  - 所谓的"自我改进"到底是什么？

- **共识观点**:
  - "自我改进"是误导性标题。它不是指模型在推理时自我进化，而是指训练过程中使用 RL 同时优化解决方案和 scaffold 的生成。这种表述引发了社区的普遍批评。
  - 该模型本质上是基于 Qwen 3.5 和 Gemma 4 的强监督微调/强化学习后训练版本。多位用户称其为"刷榜大师"（benchmaxxed），即刻意优化基准测试分数。
- **实际使用反馈**:
  - **正面**: 有用户测试发现 Ornith-1.0 35B 与 Qwen 3.6 35B 相比更好的地方不是得分，而是生成速度——因为其思维链更短，响应可快 3 倍。在 C++ 代码库的修改任务中感觉更实用。
  - **负面**: 多位用户报告在长会话或复杂工具调用中，模型仍然会出现幻觉和工具调用参数错误。本地 LLM 社区有用户明确表示不推荐使用 Ornith，建议直接使用 Qwen 3.6 或 3.5 122B。
- **基准测试质疑**:
  - 一位社区基准测试者（swelljoe）发现 Ornith-1.0 在"Will it Mythos?"测试中表现很差，仅找到了几乎所有模型都能找到的同一个 bug，并且在不使用工具时严重幻觉。但用户也指出在该测试中，Kimi K2.6/K2.7 等模型排名同样异常，可能反映的是基准测试框架的问题而非模型本身。
- **工程经验**:
  - 这是首个未被本地 LLM 社区立即拒绝的 Qwen 微调版本，但社区接受度仍然有限。
  - 用户发现，利用这些 Agent 模型的关键在于提示词工程和参数调整。模型对 prompt 结构和采样参数（推荐的 `temperature=0.6`, `top_p=0.95`, `top_k=20`）非常敏感。
  - 文件中提到 31B 密集模型，但未提供基准或权重，引起用户困惑。

## 今日洞察

LongCat-2.0 的发布释放了一个比模型性能本身更重要的信号：美团在 5 万+ 非 NVIDIA AI ASIC 上完成了 1.6T 参数 MoE 训练。这不是实验室原型，而是中国互联网公司实实在在的工程成果。这意味着出口管制试图切断的高端 GPU 供给，正在被替代硬件路径绕过——至少在中国头部企业层面，算力依赖正从“单供应商”转向“多路径”。二阶影响是：NVIDIA 的定价权边际被削弱，而开源模型生态将更加去中心化，不再受限于同一套硬件栈。对于 Anthropic、OpenAI 等依赖高端 GPU 训练的公司来说，这并非迫在眉睫的威胁，但路径分化已经开始。

OmniRoute 一天内新增 600+ stars，聚合 236 个 AI 提供商并内置智能回退与 token 压缩，这与 Google agents-cli 提供的 Agent 脚手架、评估链路、部署基础设施工具有一定的互补性。反映出 Agent 开发者的关注点正从“哪个模型最强”转向“如何用最少成本、最可靠地调度多个模型”。当开源路由网关和 CLI 工具链开始成型，Agent 产品的核心竞争力将从 prompt 工程转移到 infra 层——如同 Web 开发的演变。这对 Cursor、Claude Code 等工具是利好，但也会让模型层本身更商品化。

HN 上 Qwen 3.6 27B 的讨论（970 pts）揭示了本地模型的真实分水岭：开发者不再质疑“能否运行”，而是争论“是否值得为它买 128GB MacBook”。多位用户报告旧代码库修改效果远弱于 Claude——这印证了本地模型在“从头写”场景尚可、但在“理解既有工程”时仍是短板。长期看，随着硬件成本下降（3090 二手 $1k 也能跑），本地模型会蚕食部分云推理需求，但开发型 Agent 的核心工作流仍依赖高端云 API，除非模型架构或上下文管理有根本突破。
