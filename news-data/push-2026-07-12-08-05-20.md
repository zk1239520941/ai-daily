### 1️⃣ 智谱发布“摸高计划”：未来两年聚焦四大技术方向，不重商业变现

* **核心发布**：智谱创始人唐杰发布内部信《巨浪已来》，宣布“Touch High（摸高）”计划：未来两年不把重心放在商业变现上，集中资源投入四个技术方向，冲下一代模型。
* **四大方向**：长程任务（模型工作单位从“回答一个问题”变成“完成一个项目”）、自治智能体系统（记忆、持续学习、自我评判正在被工程消解）、自我进化（模型自己写代码、自己合成数据、自己训练自己）、安全治理（计划投入百亿级资源做机械可解释性）。
* **行业背景**：信中提到 Google DeepMind 的《From AGI to ASI》报告，指出即便单个模型能力永远停在人类水平，一亿个共享同一底层大脑的实例在群体层面已是超级智能。
* **历史验证**：智谱在 2025 年初曾将资源收敛到 Coding 和 Reasoning，从 GLM-4.5 到 GLM-5.2 在多项公开测评中摸到海外最前沿模型的能力边界。
 🔗 [Datawhale 报道](https://mp.weixin.qq.com/s?__biz=MzIyNjM2MzQyNg==&mid=2247724162&idx=1&sn=841c53ffba0c3a88ceba3e0ba52afdca)

### 2️⃣ GPT-5.6 Sol 在 Code Arena: Frontend 登顶，与 Claude Fable 5 并列第一

* **核心发布**：lmarena.ai 宣布 OpenAI 的 GPT-5.6-sol 在 Code Arena: Frontend 排名第一，与 Claude Fable 5 并列。这是 OpenAI 模型首次在 Code Arena 中达到榜首位置。
* **性能与成本**：GPT-5.6-Sol-xHigh 得分 1636，混合成本 $23.75/M，比 Claude Fable 5 便宜约 40%。在数据与分析、品牌营销、消费产品和游戏领域均排名第一。
* **行业意义**：这标志着 OpenAI 在 Agentic 编码和前端开发领域取得了重大进展，从 GPT-5.5-xhigh 的第 18 名跃升至第 1 名，证明了其在成本效益上的竞争力。
 🔗 [lmarena.ai 推文](https://x.com/arena/status/2075672492312768683) | [GPT-5.6-Sol 成本数据](https://x.com/arena/status/2075672496096047147)

### 3️⃣ Meta 发布 Muse Spark 1.1：以低成本重塑性价比前沿

* **核心发布**：Meta 正式发布 Muse Spark 1.1，通过新的 Meta Model API 和 Meta AI 使用。Alexandr Wang 称其为具备行业竞争力的 Agentic 和编程模型，在多个 Agentic 评测中可与 GPT-5.5 和 Opus 4.8 相抗衡。
* **性能数据**：在 Text Arena 的 15 个选定类别中，Muse Spark 1.1 在 12 个类别上取得提升，专家排名从 #36 升至 #15，指令遵循从 #26 升至 #5。在 Code Arena: Frontend 排名第 9，数据与分析领域排名第 2。混合成本仅 $3.5M。
* **行业意义**：Meta 的 Muse Spark 1.1 以极低成本提供了接近前沿的性能，为开发者提供了高性价比的模型选择，特别是在数据分析和内容创作领域。
 🔗 [Meta 官方博客](https://ai.meta.com/blog/introducing-muse-spark-meta-model-api/) | [Text Arena 推文](https://x.com/arena/status/2075642316027806148) | [Code Arena 推文](https://x.com/arena/status/2075642308587110616)

### 4️⃣ GPT-5.6 Sol Ultra 一小时解开 50 年数学猜想，64 个子 Agent 并行证明

* **核心发布**：Ethan Knight 宣布，GPT-5.6 Sol Ultra 在不到一小时内，使用 64 个子代理产生了 Cycle Double Cover 猜想的证明。该猜想已有 50 年历史，是图论中的一个重要未解决问题。
* **技术细节**：使用 700 词 Prompt 驾驭 64 个子 Agent，每个子 Agent 获得独立上下文并行工作，主 Agent 负责调度和评判。
* **行业意义**：这展示了前沿模型在科学发现中的潜力——通过多 Agent 并行协作，模型能够在短时间内完成需要大量推理和探索的复杂数学证明任务，标志着 AI 在数学研究中的应用进入新阶段。
 🔗 [Ethan Knight 推文](https://x.com/__eknight__/status/2075643450196971805) | [Marc Andreessen 转发](https://x.com/pmarca/status/2075702699774128280)

### 5️⃣ LangChain 发布 OpenWiki 0.1.0：为 AI 智能体装上主动记忆

* **核心发布**：LangChain 发布 OpenWiki 0.1.0，为 AI 智能体提供主动记忆能力。自动抓取 Gmail、Notion、Git 仓库、Twitter/X、Hacker News、网页搜索六类信源，将信息整理成本地 Markdown wiki 供 Agent 随时调用。
* **技术细节**：OpenWiki 靠本地定时任务保持 wiki 常新，无需常驻服务器进程。wiki 存在用户自己的机器上，刷新方式与本地工具一样：到点跑一次，把新信息拉回来，跑完就退出。
* **行业意义**：这解决了 Agent 长期记忆的核心难题——如何在不依赖常驻服务的情况下，让 Agent 拥有持续更新的外部知识库。

### 6️⃣ 美团 LongCat 开源 VitaBench 2.0：首个长期动态用户建模智能体评测基准

* **核心发布**：美团 LongCat 团队正式开源 VitaBench 2.0，这是首个真实生活场景下面向长期动态用户建模的智能体评测基准，系统性地评测大语言模型在长期、真实、动态的用户互动中个性化与主动性的能力。
* **核心数据**：包含 56 名真实特征用户、819 个复杂任务、超 2000 个动态偏好及 66 个可执行工具。平均每位用户包含 2093 个交互事件，平均时间跨度长达 1580 天。
* **核心洞察**：即使最强模型 Claude-Opus-4.6 的平均分也刚过 0.5；开启思考模式在个性化任务上并不总是有帮助；AI 普遍缺乏主动提问的意愿；即便把真实偏好直接告诉模型，多数模型仍然失败。
* **行业意义**：VitaBench 2.0 推动了评测范式从单点任务到长期陪伴、从被动执行到主动沟通的演进，揭示了个性化已是当前 Agent 的最大瓶颈。
 🔗 [美团技术团队博客](https://tech.meituan.com/2026/06/29/LongCat-VitaBench-2.0.html) | [GitHub](https://github.com/meituan-longcat/vitabench-2.0) | [项目主页](https://vitabench2.github.io/)

### 7️⃣ 美团 LongCat 开源 WBench：首个交互式视频世界模型多轮评测基准

* **核心发布**：美团 LongCat 团队提出 WBench，首个面向交互式视频世界模型的系统性多轮评测基准，覆盖 289 个测试案例和 1058 个交互轮次。
* **核心发现**：不存在全能模型，不同模型各有专长；导航是一项独立的技能，与视频画质基本无关；多轮交互是核心难点，所有模型在连续交互后表现都会变差，导航能力平均分下降 33 点；开源模型 HY-World 1.5 的导航能力在所有模型中突出。
* **可靠性验证**：WBench 的自动评分结果与 400 名人类标注者的偏好判断高度一致（Spearman ρ ≥ 0.94）。
* **行业意义**：WBench 实现了对不同输入范式模型（文本指令、相机位姿、键盘按键）的统一评测，打破了不同技术流派之间的壁垒。
 🔗 [美团技术团队博客](https://tech.meituan.com/2026/06/12/LongCat-WBench.html) | [GitHub](https://github.com/meituan-longcat/WBench) | [项目主页](https://meituan-longcat.github.io/WBench/)

### 8️⃣ 美团 LongCat 开源 General 365：通用推理评测新标尺

* **核心发布**：美团 LongCat 团队正式发布 General 365，一个将背景知识限定在 K-12 水平的通用推理评测基准，显式解耦推理能力与专业知识。
* **核心数据**：365 道原创种子题目及 1095 个扩展变体，全面覆盖复杂约束、分支与枚举、时空推理、递归与回溯、语义干扰、隐式信息、最优策略、概率与不确定性八大挑战类型。
* **实测结果**：Gemini 3 Pro 以 62.8% 的准确率艰难夺冠，绝大多数模型未能触及 60 分及格线。语义干扰与最优策略成为主要性能洼地。
* **行业意义**：General 365 将推理评测从专业知识依赖中剥离出来，直观地揭示了大模型在真实世界通用推理任务上的短板——一个能解出 IMO 难题却回答不出「走路洗车」的模型，还不能被称为真正的智能。

### Stop Telling Me to Ask an LLM

110 pts · 60 comments · [yaelwrites.com](https://blog.yaelwrites.com/stop-telling-me-to-ask-an-llm/)

**📌 内容总结**

- 作者在寻求资深人士的行业经验时，对方建议“问 Claude”。作者此前已花数小时与 LLM 交互，但问题仍未解决。她认为“问 LLM”成了“我不知道”或“我没时间”的礼貌替代，但拒绝提供个人经验。
- HN 关注点:
  - 提问者是否展示了“已付出努力”的证据，是决定是否被回复“问 LLM”的关键。
  - “问 LLM”与过去的“LMGTFY”在社交功能上高度相似，都是对低质量问题的防御性回应。
  - 资深人士是否真的认为 LLM 能比自己回答得更好，还是单纯想节省时间。

**💬 讨论总结**

- **共识观点：** 提问质量决定回复质量。如果提问者展示了自己已做的研究（“我试了A、B、C，A更好但C最快”），资深人士通常不会直接回复“问 LLM”。反之，低努力问题得到低努力回复是合理的。
- **工程经验：** 在代码审查中，开发者直接粘贴 LLM 对审查意见的回复，而不展示自己的理解，被视为不尊重且无效的沟通。技术负责人无法判断开发者是否真正理解了代码或反馈。
- **反对意见：**
  - 部分资深人士认为，30 年经验教会他们的正是“LLM 能比我回答得更好”，因此“问 Claude”是真诚的建议，而非敷衍。
  - 有观点认为，频繁被打断的资深员工有权利使用任何工具（包括 LLM）来减少上下文切换成本，这是对自身时间的合理保护。
  - 有人指出，作者可能没有意识到，对方给出“问 Claude”是因为对方自己也忘记了那个“30 年经验”的答案。

### Mesh LLM: distributed AI computing on iroh

26 pts · 10 comments · [iroh.computer](https://www.iroh.computer/blog/mesh-llm)

**📌 内容总结**

- 作者想做什么：构建一个开源分布式推理系统，将用户已有的 GPU 和内存（笔记本、GPU 工作站、服务器等）组成 P2P 网格，暴露为 OpenAI 兼容 API。支持模型本地运行、路由到对等节点、或跨多机流水线拆分（Skippy 模式）。
- HN 关注点:
  - 跨网络推理的延迟和吞吐量是否实用。
  - 与现有方案（如 cocompute.ai）的差异。
  - 安全性：节点间通信是否加密。

**💬 讨论总结**

- **共识观点：** 跨网络推理的吞吐量是主要瓶颈。消费者网络（甚至 10GbE）相比本地显存/内存慢得多。作者在模型列表中提到 Qwen 235B MoE 在 2 节点上达到 16 tok/s，但未说明节点配置和网络条件。
- **工程经验：** 性能可粗略估算：自回归解码阶段，每 token 需传输约 `2 * hidden_size * num_shards` 字节。预填充阶段则除以 chunk size。
- **风险/限制：** 对于交互式使用，16 tok/s 接近但未完全达到舒适体验。网络延迟和带宽是硬约束。
- **支持观点：** iroh 的核心优势是“dial-a-key”加密 QUIC 连接，节点间通信默认加密，解决了安全问题。
