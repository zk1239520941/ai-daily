### 2️⃣ Anthropic将Claude Code系统提示缩减80%：模型越强，指令越少

* **核心发布**：Peter Yang在播客中采访Claude Code团队成员Thariq，透露Anthropic近期将Claude Code的系统提示缩减了80%。Thariq解释称：“随着模型变得更智能，它们需要更少的指令、更少的约束和更少的示例。示例反而会限制模型，因为它会认为‘你想要像这个示例一样的东西’。去掉示例后，模型可以更自由地发挥。”
* **技术逻辑**：这一做法背后的逻辑是，更强的模型需要更多的“运行空间”来发挥其能力，过长的系统提示反而会限制模型的创造性。团队建议用户在发布新模型时主动裁剪上下文。
* **行业意义**：这标志着AI Agent开发从“提示工程”向“模型能力信任”的转变——当模型足够强大时，开发者应减少人为约束，让模型自主探索更优解。

### Claude Code uses Bun written in Rust now

393 pts · 556 comments · [simonwillison.net](https://simonwillison.net/2026/Jul/19/claude-code-in-bun-in-rust/)

**📌 内容总结**

- 作者通过 `strings` 命令在本地 Claude Code 二进制中发现了 Rust 源码路径（如 `src/bundler/bundle_v2.rs`）和 Bun v1.4.0 版本号，证实 Claude Code 确实使用了 Rust 重写版的 Bun。该版本号高于公开的最新 Bun 1.3.14，说明 Anthropic 在打包一个尚未正式发布的 canary 版本。
- HN 关注点:
  - 这是否意味着 Bun 项目已实质上被 Anthropic 控制，开源治理结构成谜。
  - AI 驱动的 100 万行 Zig→Rust 重写仅用一个月就进入生产环境，对软件工程意味着什么。
  - 用户端几乎无感知（Linux 启动快 10%），但内部工程代价和安全性存疑。

**💬 讨论总结**

- **共识观点**：用户端几乎未感知到变化，“boring is good” 是事实。多数 Claude Code 用户表示没有注意到任何异常。
- **工程经验**：
  - 重写动机主要是解决 Zig 版本中的内存安全问题（手动管理生命周期导致的 bug 累积），而非性能。Rust 的编译期检查为 AI 生成的代码提供了确定性验证。
  - 这次重写本质上是“逐行翻译”（transpilation），生成的 Rust 代码并不符合 Rust 惯用法，大量使用 `unsafe`。有评论指出这并未真正获得 Rust 的内存安全保障。
  - 有经验的开发者指出，Zig/C 本身可以通过 arena 分配等模式写出健壮代码，但 LLM 倾向于使用“托管语言风格”的小对象分配模式，这恰好是 Zig 的弱点。
- **历史背景**：Bun 最初由 Go 重写为 Zig，现在又由 Zig 重写为 Rust。这是 Bun 的第二次重大语言迁移。
- **商业现实**：Anthropic 收购了 Bun（Oven.sh），这笔交易的成本（$145,000 的 Claude 计算时间）和后续治理模式引发关注。有评论指出，Claude Code 作为一个 TUI 应用，其供应链长到需要收购一个运行时，本身就反映了工程选型的异常。
- **风险/限制**：
  - 代码可审计性：100 万行 AI 生成的代码几乎不可能人工审计，存在供应链安全风险。
  - Claude Code 近期出现更多 TUI 渲染问题（窗口 resize 后内容乱码、segfault），部分用户怀疑与 Rust 重写有关。
  - 有用户报告 Claude Code 会尝试用 `sudo pacman` 安装 Bun，行为异常。
- **反对意见**：

### OpenAI reduces Codex Model Context Size from 372k to 272k

319 pts · 148 comments · [github.com/openai](https://github.com/openai/codex/pull/33972/files)

**📌 内容总结**

- 这是一个将模型元数据回退到 Codex 0.144 版本的 PR，将默认上下文窗口从 372k tokens 缩减至 272k tokens。OpenAI 员工 Tibo 在 X 上解释这是临时措施，后续会恢复。
- HN 关注点:
  - 上下文缩减的真实原因：成本/延迟权衡，而非能力限制。372k 的 token 成本比 272k 高出约 87%（二次注意力机制）。
  - Codex 的自动压缩（compaction）机制是否足以弥补窗口缩小带来的损失。
  - 该 PR 还新增了系统提示，要求模型在执行破坏性操作前进行只读检查——这是对 Codex 曾误删用户家目录的修复。

**💬 讨论总结**

- **共识观点**：大多数用户认为 272k 对日常编码足够，但重度用户（处理大型代码库、多轮迭代）会感受到明显差异。
- **工程经验**：
  - 多位有经验的用户表示，模型在超过 150k-250k 后质量会显著下降（“dumb zone”），因此主动限制上下文反而能获得更好结果。
  - 好的项目文档（`AGENTS.md`、`plan.md`）和模块化代码设计比大上下文窗口更重要。有用户通过规则文件（60-80k tokens）指导代理行为，但建议用 LLM 优化这些规则文件以减少 token 消耗。
  - 自动压缩的问题：压缩后丢失细节，且无法回退。有用户报告压缩后模型会忘记正在执行的任务，导致死循环（读取→压缩→再读取→再压缩）。
- **历史背景**：Codex 早期版本经常出现“模型上下文大小超限”错误，压缩机制也无法恢复。这个问题在几个月前才得到改善。
- **商业现实**：OpenAI 对超过 272k 的长上下文收取 100%/50% 的溢价，缩减窗口直接降低了用户的 API 成本。有评论指出这是成本控制措施。
- **风险/限制**：
  - 自动压缩无法禁用，且触发阈值在 10-20% 左右，实际可用上下文只有约 218k。
  - 压缩后模型可能过度关注旧的 steering message，导致行为偏差。
  - 有用户指出，Codex 正在走向“对用户隐藏尽可能多信息”的方向（加密子代理提示、不展示压缩摘要内容），这降低了可调试性。
- **反对意见**：
  - 部分用户认为上下文大小根本不重要，因为好的工作流（分而治之、子代理、计划文件）可以在 200k 内完成所有任务。
  - 有用户指出，即使有 1M 上下文，模型在 50% 后质量也会下降，不如主动 /clear 重新开始。
