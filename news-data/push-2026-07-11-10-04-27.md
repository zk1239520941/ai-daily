### 2️⃣ Apple 起诉 OpenAI 系统性窃取商业机密，指控涉及硬件业务

* **核心发布**：Apple 向加州北区联邦地方法院提起诉讼，指控 OpenAI 系统性窃取苹果商业机密，用于开发自己的 AI 硬件设备。被告包括 OpenAI、硬件负责人 Tang Tan（前苹果高管）等。
* **具体指控**：诉状称 OpenAI 的硬件业务“建立在最不稳固的基础上，其核心是非法依赖盗用的商业机密”。Tang Tan 被指控在面试中利用苹果内部代号获取机密信息，并传播内部离职文档教新员工如何规避离职安全审查。
* **行业意义**：这起诉讼发生在 OpenAI 即将 IPO 前夕，如果指控成立，将严重打击 OpenAI 的硬件计划。Gary Marcus 评论称这是“建立在被盗 IP 和谎言之上的公司涉嫌窃取更多 IP”。

### Apple sues OpenAI, accuses ex-employees of stealing trade secrets

542 pts · 255 comments · [site](9to5mac.com)

**📌 内容总结**

- 苹果在加州联邦法院对 OpenAI、两名前员工（Chang Liu、Tang Yew Tan）及 OpenAI 收购的硬件公司 io Products 提起诉讼，指控其系统性窃取苹果硬件相关的商业秘密。
- HN 关注点:
  - 诉状细节：Liu 离职后利用未归还的苹果笔记本和认证漏洞，持续访问并下载了数千页机密硬件工程文件；Tan 在面试苹果在职员工时，要求其携带“实际零件”和“CAD/设计工件”进行“展示与讲解”，并使用苹果内部项目代号询问未发布产品计划。
  - OpenAI 被指系统性指导新员工规避苹果离职安全审查，包括共享苹果内部“Need to Know”安全流程文档，并建议员工不要告知苹果其去向。
  - 苹果声称 OpenAI 利用窃取的信息，欺骗一家供应商为其执行一项特定的“金属精加工”商业秘密技术。

**💬 讨论总结**

- **共识观点**：诉状中的具体行为（如利用漏洞下载文件、要求携带实物零件面试）远超普通的人才流动或竞业限制范畴，属于明确的商业秘密盗窃。多数评论认为，如果指控属实，OpenAI 的行为在法律和商业道德上都站不住脚。
- **工程经验与历史背景**：多位评论者指出，大型科技公司（尤其是苹果）对数据外泄有极其严密的监控系统（日志、网络访问记录），高薪工程师试图窃取数据的行为“愚蠢至极”。有评论类比了苹果过去起诉三星的案例，但认为本案性质更严重，因为涉及的是未发布的内部机密，而非已上市产品的设计模仿。
- **商业现实与风险**：普遍认为此案对 OpenAI 的硬件计划和即将到来的 IPO 构成重大威胁。评论推测，苹果选择此时起诉，可能意在 OpenAI 上市前增加其法律负债。多数人预测案件将以巨额和解告终，而非进入全面证据开示程序。
- **反对/质疑意见**：少数评论质疑“金属精加工技术”等知识是否应被视为商业秘密，认为员工将知识带往新公司是正常的市场流动。但这些观点在讨论中被明显反驳，理由是本案涉及的是有组织的、主动的窃取行为，而非员工利用个人经验。

### GPT-5.6 Sol Ultra produces proof of the Cycle Double Cover Conjecture [pdf]

353 pts · 281 comments · [site](cdn.openai.com)

**📌 内容总结**

- OpenAI 宣布其模型 GPT-5.6 Sol Ultra 在约一小时内，自动生成了图论中著名的“圈双覆盖猜想”（Cycle Double Cover Conjecture）的证明。该猜想已悬而未决约50年。
- HN 关注点:
  - 证明本身仅3页，非常简洁，且未使用近30年内的新数学工具，而是巧妙地组合了已有的定理（如8-流定理）。
  - OpenAI 同时发布了模型的输入提示词（prompt），其中包含大量元指令，如“拒绝模糊的进展报告”、“假设完整的肯定性证明存在”等，用于引导模型不轻易放弃。
  - 社区核心争议在于：该证明是否已被验证为正确？目前仅由 OpenAI 单方面发布，尚未经过同行评审或形式化验证（如 Lean 证明助手）。

**💬 讨论总结**

- **共识观点**：无论证明最终是否完全正确，这都是一项里程碑式的成就。模型能够自主提出一个简洁、有洞察力的证明思路，表明其在数学推理上达到了新高度。许多评论认为，这标志着 AI 在数学领域从“辅助工具”向“独立研究者”的转变。
- **工程经验与历史背景**：有评论指出，该证明的简洁性（“利用了一个所有专家都忽略的巧妙技巧”）恰恰是 AI 的优势——它不受人类思维定势和失败经验的限制。提示词中大量“不要放弃”的指令，反映了当前模型在解决困难问题时仍需外部强制力来克服过早收敛的倾向。
- **风险与限制**：主要质疑集中在验证环节。证明未使用 Lean 等证明助手进行形式化验证，而图论领域的证明形式化基础设施尚不成熟。多位评论者强调，在没有专业数学家或形式化验证的情况下，这类“AI 生成”的证明可能包含细微但致命的错误。有评论讽刺道：“如果这个证明是错的，你怎么知道？这正是问题所在。”
- **反对/质疑意见**：部分评论认为这主要是 OpenAI 的市场营销行为，旨在展示模型能力。也有观点指出，模型可能是在大量尝试中“碰巧”找到了一个可解的猜想，存在严重的幸存者偏差。此外，有评论质疑，如果证明最终被证实有误，OpenAI 是否会撤回并承认错误。

## 今日洞察

**Apple 诉 OpenAI 的诉状细节，暴露了 AI 硬件公司获取工程知识的真实路径。** 诉状指控 Tang Tan 在面试苹果在职员工时，要求对方携带“实际零件”和“CAD/设计工件”进行“展示与讲解”，并使用苹果内部项目代号询问未发布产品计划。这不是常规的人才流动或竞业限制——它指向一种系统性的、有组织的硬件知识转移方式。如果指控成立，OpenAI 的硬件业务（包括与 io Products 的整合）将面临根本性的合法性危机。HN 讨论中多数评论认为，此案远超普通商业秘密纠纷，涉及的是“有组织的、主动的窃取行为”，而非员工利用个人经验。这对 OpenAI 即将到来的 IPO 构成直接威胁——法律负债将直接影响估值。

**GPT-5.6 Sol Ultra 证明 Cycle Double Cover 猜想，暴露了当前 AI 数学推理的核心矛盾：生成能力远超验证能力。** 模型在一小时内用 64 个子代理生成了一个 3 页的简洁证明，利用了“所有专家都忽略的巧妙技巧”——这正是 AI 的优势，不受人类思维定势限制。但 HN 讨论的核心争议在于：该证明尚未经过同行评审或形式化验证（如 Lean 证明助手）。多位评论者强调，图论领域的证明形式化基础设施尚不成熟，AI 生成的证明可能包含“细微但致命的错误”。这揭示了一个关键约束：AI 在数学领域的应用瓶颈正从“能否生成”转向“能否验证”。OpenAI 选择此时发布未经验证的证明，本质上是 PR 行为——展示模型能力，而非解决数学问题。

**GPT-5.6-sol 在 Code Arena: Frontend 登顶，但更值得关注的是成本结构的变化。** 从 GPT-5.5-xhigh 的第 18 名跃升至第 1 名，混合成本 $23.75/M，比 Claude Fable 5 便宜约 40%。这标志着 OpenAI 在 Agentic 编码领域重新获得竞争力，但竞争维度已从纯性能转向“性能/成本比”。Muse Spark 1.1 以 $3.5M 的混合成本在 Code Arena 排名第 9，进一步压缩了性价比前沿。开发者选择模型时，成本约束正在成为与性能同等重要的决策变量——这与 HN 上对 Claude Code 成本争议的讨论一致。
