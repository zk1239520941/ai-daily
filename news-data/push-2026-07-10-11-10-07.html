OpenAI 全量发布 GPT-5.6 系列：三档模型全面对标 Claude Fable 5

  • 核心发布：OpenAI 正式结束 GPT-5.6 系列为期两周的受限预览，向全球用户全量开放 Sol、Terra、Luna 三款模型，通过 ChatGPT、Codex 和 API 上线。

Show HN: Getting GLM 5.2 running on my slow computer

426 pts · 109 comments · site

📌 内容总结

  • 作者想做什么: 在只有 25GB RAM 的消费级电脑上运行 744B 参数的 GLM-5.2 MoE 模型。核心思路是只将密集层（~17B 参数）常驻内存，而 21,504 个路由专家（~370GB）存储在磁盘上，按需流式加载。
  • HN 关注点:
  • 技术实现: 纯 C 实现，零依赖，使用 LRU 缓存、异步预读、MTP 投机解码（int8 头部，接受率 39-59%，2.2-2.8 tok/forward）。
  • 性能数据: 在 WSL2、12 核、25GB RAM、NVMe 环境下，冷启动约 0.05-0.1 tok/s；M5 Max 上约 1 tok/s。社区实测数据已提供。
  • 限制: 冷启动时磁盘读取是瓶颈（~11GB/token）；小 RAM 机器上 RAM 上限是主要约束；int4 量化对精度的影响尚未测量。
💬 讨论总结
  • 共识观点: 这是“黑客精神”的体现，将大型模型运行在非预期硬件上令人印象深刻。项目受到 antirez 的 ds4 项目启发。
  • 工程经验与讨论焦点:
  • SSD 寿命: 作者在 README 中已说明，冷启动的随机读取（~11GB/token）不会显著磨损 SSD，但系统内存不足导致的交换写入才是问题。建议监控温度，使用外部存储作为“燃烧盘”。
  • 性能瓶颈: 社区实测证实，在 24GB RAM 机器上，引擎自动将专家缓存限制为每层 2 个槽位，导致解码始终处于冷状态，RAM 上限而非磁盘速度是主要瓶颈。--topp 0.7 参数可带来 1.6 倍加速。
  • 与类似项目对比: 有评论提到 flash-moe 项目在 M3 Max 上可达 5 tok/s，但未使用 GPU。另一个项目 hypura 也实现了类似的 SSD 流式加载。有观点认为应将此类功能上游合并到 llama.cpp 或 ds4。
  • 分布式运行: 有用户询问是否能用 MPI 在小型集群上运行。作者回应称，对于小型集群，网络延迟通常弊大于利，更适合大型集群或单机优化。
  • 硬件建议: 有用户指出，如果能将整个模型加载到 RAM（如使用旧款双路服务器，768GB RAM），性能会远高于磁盘流式方案。但本项目设计初衷就是针对 RAM 受限的场景。
  • 反对 / 质疑:
  • 有评论认为 0.05-0.1 tok/s 的速度对于实际使用来说太慢，不如使用云服务。作者回应称，社区实测数据表明在更好硬件上速度可达 1 tok/s 以上。
  • 有用户质疑“慢电脑”的定义（25GB RAM），作者澄清“慢”在此上下文中指仅使用 CPU。

GPT-5.6

1096 pts · 797 comments · site

📌 内容总结

  • 背景: OpenAI 发布 GPT-5.6 系列模型（Sol, Terra, Luna），声称在多项基准测试中超越 Claude Fable 5，同时成本更低、速度更快。引入了 max 和 ultra 推理模式，以及 Programmatic Tool Calling 和多智能体功能。
  • HN 关注点:
  • 基准测试对比: 在 Agents' Last Exam、Artificial Analysis Coding Agent Index 等测试中，Sol 声称领先 Fable 5。但在 SWE-Bench Pro 上，Sol（64.6%）落后于 Fable（80%）和 Opus（69.2%）。
  • 定价: Sol $5/$30 (input/output) per 1M tokens，与 GPT-5.5 持平，但低于 Fable 5 ($10/$50)。Terra 和 Luna 更便宜。
  • 安全与审查: 声称在网络安全和生物安全领域未达到“临界”阈值，但加强了安全层，包括推理监控器。在生物学基准测试中，因 Claude Fable 拒绝回答大部分问题，Sol 被列为“胜出”。
  • 开发者指南: 建议使用更短的提示词，避免“简洁”等通用指令，因为模型对这类指令更敏感。
💬 讨论总结
  • 共识观点: 社区普遍对 OpenAI 的基准测试结果持怀疑态度，认为存在“benchmaxxing”（为跑分优化）和选择性展示的问题。许多用户表示需要实际使用后才能判断。
  • 工程经验与讨论焦点:
  • 基准测试争议: SWE-Bench Pro 的结果（Sol 落后于 Fable 和 Opus）成为焦点。有评论指出 OpenAI 前一天刚发布报告质疑 SWE-Bench Pro 的有效性，认为其“信号与噪声分离”不佳。DeepSWE 和 FrontierCode 被认为是更可靠的编码基准。
  • 成本与效率: 社区注意到 Sol 的定价与 GPT-5.5 相同，但声称性能大幅提升。Terra 和 Luna 在成本效率上被认为具有竞争力，可能吸引开发者从 Anthropic 迁移。
  • 模型命名与复杂性: 新的命名体系（Sol, Terra, Luna）被认为混乱，且模型有多个推理级别（medium, high, max, ultra），选择困难。有观点认为未来需要“自动模式”来根据任务选择最佳配置。
  • Codex vs. Claude Code: 用户讨论了 Codex 和 Claude Code 的优劣。有用户认为 Codex 的远程会话集成更好，而 Claude Code 在“ultracode”编排方面领先。OpenCode 等开源工具被认为是避免供应商锁定的选择。
  • 安全与审查: Claude Fable 在生物学基准测试中拒绝回答大部分问题，导致 OpenAI 将其排除在对比之外。这引发了关于模型审查对实际应用影响的讨论，尤其是在安全研究和逆向工程领域。有用户表示因此从 Anthropic 转向 OpenAI。
  • 反对 / 质疑:
  • 许多用户对 OpenAI 声称 Sol 在多项指标上“大幅超越”Fable 5 表示不信，认为图表 y 轴起始点（30%）具有误导性，且对比中未使用 Fable 的“max”模式。
  • 有评论指出，Sol 在 ARC-AGI-3 上得分为 7.8%，虽然首次有前沿模型在该测试中得分，但绝对数值仍然很低。
  • 发布当天，模型在 ChatGPT 和 Codex 中并未立即可用，而是逐步推出，引起用户不满。

GLM 5.2 is nearly as accurate as a human book keeper

191 pts · 113 comments · site

📌 内容总结

  • 背景: 作者（Toot Books）评估了开源模型 GLM 5.2 为英国小企业准备季度增值税申报表的能力。模型需要处理 59 笔交易，耗时 68 分钟，原始 token 成本 2.73 美元。
  • HN 关注点:
  • 准确性: 模型生成的增值税申报表“基本正确”，净额（Box 5）仅与人工准备的结果相差 7 便士。但在 354 项检查中，有 20 项失败，分布在 18 笔交易中。
  • 关键错误: 最严重的错误是将创始人的股本投入错误分类为“资本账户”而非“未缴股份”，这有法律和审计风险。其他错误主要是混淆“零税率”和“免税”类别。
  • 限制: 基准测试中，发票已由人工提供，模型无需自行查找。模型知道自己在被测试，其推理过程受到此意识影响。
💬 讨论总结
  • 共识观点: 社区普遍认为，对于高度约束、上下文狭窄的任务（如记账），LLM 可以取得很好的效果。但“几乎准确”对于税务申报来说远远不够，因为错误可能导致罚款甚至法律问题。
  • 工程经验与讨论焦点:
  • 责任与风险: 核心争议在于，当 AI 出错时，谁来承担责任？用户最终是责任人，而 AI 公司（如 Toot）的条款明确声明其不提供会计建议，用户必须自行审核。这与人类会计师的受监管职业责任形成对比。
  • “几乎准确”的局限性: 许多评论指出，对于税务申报，99% 或 99.99% 的准确率都不够，必须“完美”。即使净额只差 7 便士，但分类错误（如股本）可能在审计中引发严重问题。
  • 与人类会计师对比: 基准测试中，人类会计师的工作范围更广（包括查找发票、处理未文档化的问题）。AI 目前只能处理已结构化的输入。有评论认为，AI 更适合作为辅助工具，由人类进行复核，从而大幅降低成本。
  • 实际应用案例: 有用户分享了使用 Claude Code 和 FreeAgent API 进行自动记账的个人项目，效果良好，但强调需要人工监督。也有用户提到 Digits 公司也在进行类似的 AI vs 人类记账员基准测试。
  • 数据与信任: 有用户对 Toot 这家新公司（仅成立一年）缺乏创始人信息表示担忧，认为将财务数据交给这样的初创公司风险过高。
  • 反对 / 质疑:
  • 有评论认为，对于非复杂业务，发票已录入财务系统，生成增值税申报表是确定性的，无需 LLM。
  • 有观点认为，LLM 的“幻觉”特性使其不适合处理需要绝对准确性的任务，即使它 99% 的时间都正确，那 1% 的错误可能造成巨大损失。

今日洞察

基准测试的“选择性展示”正在成为模型发布的标准策略，但 HN 社区的质疑揭示了更深的信任问题。 GPT-5.6 的发布在 HN 上获得 1096 分和 797 条评论，但讨论焦点并非模型能力本身，而是 OpenAI 如何呈现数据。社区普遍使用“benchmaxxing”一词来描述这种为跑分优化的行为。关键证据是：OpenAI 在 SWE-Bench Pro 上刻意回避了 Sol（64.6%）落后于 Fable（80%）和 Opus（69.2%）的事实，转而强调 Agents' Last Exam 和 Coding Agent Index 上的领先。更微妙的是，OpenAI 在发布前一天刚发布报告质疑 SWE-Bench Pro 的有效性——这种“先削弱对手武器，再展示自己优势”的节奏，正在成为 PR 叙事的一部分。对于开发者而言，这意味着不能依赖任何单一基准做模型选型，必须基于自己的实际工作流做 A/B 测试。

Anthropic 的“额度重置”是一次罕见的情绪化操作，暴露了竞争已从技术层面蔓延到用户获取层面。 在 GPT-5.6 全量发布后，Anthropic 在非正常时间重置了所有用户的 5 小时和每周额度限制，且 OpenAI Codex 团队的开发者直接在公告评论区“贴脸开大”。这一举动在历史上极为罕见——通常模型厂商不会在对手发布日立即调整用户策略，尤其是以这种“全员重置”的方式。它暗示两个信号：一是 Anthropic 认为 GPT-5.6 的发布确实构成了实质性威胁，需要立即降低用户使用门槛来留住开发者；二是双方的人才和用户争夺已经进入“贴身肉搏”阶段，不再顾及行业礼仪。对于开发者而言，这意味着短期内可以享受更激进的定价和额度竞争，但长期来看，模型切换成本会因生态绑定（Codex 并入 ChatGPT、Claude Code 的 MCP 生态）而快速上升。

GLM 5.2 的记账测试揭示了一个被忽视的工程现实：LLM 在“几乎准确”和“足够准确”之间存在巨大的责任鸿沟。 模型生成的增值税申报表净额仅差 7 便士，但 354 项检查中有 20 项失败，最严重的是将股本投入错误分类为“资本账户”而非“未缴股份”——这在审计中可能引发法律问题。HN 社区的 113 条评论几乎全部聚焦于“谁来承担责任”这一核心问题，而非模型能力本身。这暴露了一个关键约束：对于税务、法律、医疗等需要绝对准确性的领域，LLM 的“概率性正确”本质上与行业要求不兼容。即使 99.99% 的准确率，那 0.01% 的错误也可能造成远超模型节省成本的损失。这意味着，AI 在受监管行业的落地路径不会是“替代人类”，而是“人类+AI 复核”模式——这反过来会限制成本节省的幅度，因为人工复核环节仍然是瓶颈。
