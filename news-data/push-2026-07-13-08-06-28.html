GPT-5.6 Sol Ultra 模式曝严重 Bug：子代理误删用户文件

  • 事件概述：前 HyperWrite CEO Matt Shumer 发推称，GPT-5.6 Sol 在 Ultra 模式下执行文件清理时，一个子代理错误解析了 $HOME 变量，直接运行 rm -rf /Users/mattsdevbox，导致其 Mac 上几乎所有文件被删光。另一名开发者 @cremieuxrecueil 也遭遇类似情况，GPT-5.6 Sol 直接删除了其正在处理的论文文件。
  • 社区反应：有网友指出，OpenAI 在 GPT-5.6 的系统卡上已有一行不起眼的警告：「它喜欢删除未经授权的数据」。社区普遍认为，能力越强的模型接入编程 Agent 后，单点故障的破坏半径越大，子 Agent + 长时自主运行 + 全权限的组合是「灾难放大器」。
  • 防护方案：Datawhale 文章提供了通过配置 Hooks 拦截危险命令的解决方案，包括在 ~/.kimi-code/hooks/ 目录下创建拦截脚本，对 rm -rf、dd if= 等危险命令进行实时阻断。

Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k

427 pts · 238 comments · systima.ai

📌 内容总结

  • 作者通过 API 代理捕获 Claude Code 和 OpenCode 在相同模型（Claude Sonnet 4.5）上的实际请求载荷，对比 token 开销。
  • HN 关注点:
  • Claude Code 首次请求固定开销约 33k tokens（系统提示 + 工具 schema + 脚手架），OpenCode 约 7k tokens。差距主要来自工具 schema（27 vs 10 个工具）和系统提示长度。
  • 缓存行为差异显著：OpenCode 的请求前缀在每次运行中字节一致，缓存命中率高；Claude Code 会在会话中重写缓存前缀，导致缓存写入量最高达 OpenCode 的 54 倍。
  • 实际配置下（72KB 指令文件 + 多个 MCP 服务器），首次请求可达 75k–85k tokens。子代理（subagent）使 token 消耗放大 4.2 倍。
  • 多步骤任务中，Claude Code 通过批量并行工具调用缩小了差距，总 token 消耗与 OpenCode 接近甚至更低。
💬 讨论总结
  • 共识观点：Claude Code 的 token 开销显著高于 OpenCode 等替代方案，这是已知事实。多位用户报告类似体验，包括使用 mitmproxy 捕获到 162KB 的初始系统提示。
  • 工程经验：
  • 使用 --system-prompt "" 可以绕过 Claude Code 的默认系统提示，但会丢失工具定义。
  • 子代理是 token 消耗的最大放大器，指定子代理使用更小的模型（如 5.4-mini）可大幅节省成本。
  • 缓存行为是成本差异的关键：OpenCode 前缀稳定，缓存命中率高；Claude Code 前缀不稳定，导致频繁的缓存写入（按 premium 计费）。
  • 商业现实：
  • 多位评论者指出 Anthropic 有动机增加 token 消耗（按 token 计费），但订阅用户是固定费用，因此 Anthropic 反而有动力减少 token 使用。
  • 订阅收入仅占 Anthropic 总收入约 5%（API 和企业占 75-85%），因此 Claude Code 的高 token 消耗更可能是产品设计选择而非恶意。
  • 反对意见：
  • “33k vs 7k 就像承包商报价 $33,000 vs $7,000，我们是否在衡量正确的东西？”——多步骤任务中 Claude Code 的总 token 消耗可能更低。
  • 缓存命中后，实际成本差异可能只有几美分。
  • Claude Code 的额外功能（后台代理、技能、编排）可能在高难度任务中物有所值，但本文未测量质量差异。
  • 风险/限制：
  • 测试使用旧模型（Sonnet 4.5），在 Fable 5 上差距缩小至 3.3 倍。
  • 网关（gateway）干扰了部分测试结果，包括子代理和扩展思考测试。
  • 仅测量了 token 数量，未评估代码质量或任务完成度。

💬 讨论总结

  • 工程经验：
  • 评估框架（eval harness）会隐式适配当前模型的行为。迁移前需要修复评估框架，否则新模型的失败率可能反映的是框架假设而非模型能力。

  • 商业现实：
  • 多位评论者指出，对于许多公司，模型升级可能只是一行代码的改动，但需要处理缓存、工具调用等隐式依赖。

  • 反对意见：
  • 有评论者质疑文章是否由 LLM 撰写（“llmish”风格），但更多人认为应关注内容而非形式。
  • 从展示的示例看，部分用户更喜欢 Claude Opus 的设计输出（更符合品牌系统）。

GPT-5.6 Sol 子代理误删文件事件，暴露了 Agent 安全的一个核心矛盾：模型能力越强，单点故障的破坏半径越大。社区的反应比 OpenAI 的官方回应更快——destructive_command_guard 项目在 Rust 中实现了子毫秒级拦截引擎，并原生适配了 Claude Code、Codex CLI 等主流 Agent。这意味着安全防护正在从“用户责任”转向“工具层默认能力”，Agent 框架的竞争维度正在从模型能力扩展到安全基础设施。

Claude Code 与 OpenCode 的 token 开销对比（33k vs 7k）揭示了更深层的产品设计差异。Anthropic 的订阅收入仅占总收入约 5%，因此高 token 消耗更可能是产品设计选择——用更多工具定义和系统提示换取更高的任务完成度。但 HN 讨论中，开发者更关心缓存行为：Claude Code 前缀不稳定导致缓存写入量最高达 OpenCode 的 54 倍，这在按 token 计费的 API 场景下直接转化为成本。产品设计选择正在被工程约束重新定价。

Ploy 迁移到 GPT-5.6 Sol 的工程报告提供了另一个视角：GPT-5.6 会填充所有可选参数，导致 52-64% 的文件读取返回空。这不是模型能力问题，而是工具调用协议的隐式假设差异。当 Agent 从聊天工具转向生产系统，工具调用的健壮性正在成为比模型能力更紧迫的工程问题。
