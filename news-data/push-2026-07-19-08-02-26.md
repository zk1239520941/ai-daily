### 3️⃣ 港科大发现 MoE 路由脆弱性：重复 token 可致负载不均，TTFT 最高增加 148%

* **核心发布**：来自港科大的研究团队在 ICML 2026 发表论文，提出 RepetitionCurse——一种针对 MoE 大模型服务的黑盒压力测试方法。该方法利用高度重复的输入模式，诱导专家路由将大量 token 路由到同一小批专家上。
* **攻击原理**：当输入中出现高度重复的 token 模式时，连续 token 的隐藏表示变得非常相似，导致 router 反复将它们分给同一批专家。在 Expert Parallelism 部署下，这会造成单张 GPU 成为 straggler，其他 GPU 空等同步。
* **攻击效果**：在常见 8-GPU 部署上，RepetitionCurse 可使 MoE 模型的 TTFT（首 token 延迟）增加 20% 到 148%。EP size 越大，攻击效果越明显。攻击请求还可“搭车”影响同一 batch 中的正常用户请求。
* **行业意义**：该研究揭示了 MoE 模型在服务系统层面的安全漏洞，提醒服务商需监控 per-expert 和 per-GPU 的实时负载，并对极端低熵请求进行隔离或降级。
 🔗 [新智元报道](https://mp.weixin.qq.com/s?__biz=MzI3MTA0MTk1MA==&mid=2652713777&idx=3&sn=01c4801592c61a1f52ae653764b5ac39) | [论文链接](https://arxiv.org/abs/2512.23995)

### Setting up your spare Mac for Claude Code to control, a step-by-step guide

166 pts · 125 comments · [ykdojo.github.io](ykdojo.github.io)

**📌 内容总结**

- 作者想做什么：让 Claude Code 完全控制一台备用 Mac，实现远程委托任务（开发、研究等），同时隔离风险。
- 解决的问题：在容器中运行 Claude Code 存在网络隔离不彻底、无法运行 GUI 应用（如 Unity）等限制。
- 技术实现：通过 SSH、tmux、LaunchAgent 等工具，在备用 Mac 上创建一个隔离的、可远程控制的 Claude Code 环境，并支持 computer use（屏幕截图、鼠标键盘控制）。
- 产品设计：提供 `ic` 脚本作为统一入口，支持从手机（通过 Claude App 的 Remote Control）和主 Mac 控制。
- 用户反馈焦点：评论区主要讨论“为什么需要这样做”以及“替代方案”。

**💬 讨论总结**

- **共识：对于需要 GUI 或完整 macOS 能力的场景，物理隔离是合理的。** 容器（如 Dispatch/Cowork）在权限和 GUI 支持上存在限制，VM（如 UTM）缺乏图形加速，物理机是唯一能运行 Unity 等应用的方案。
- **工程经验：macOS 权限模型是主要障碍。** Screen Recording 和 Accessibility 权限必须由人类在 GUI 中手动授予，且授予对象是 `tmux` 而非 `claude`。Full Disk Access 可抑制“访问其他应用数据”的弹窗。
- **商业现实：Anthropic 的定价模型复杂且昂贵。** 有评论者指出，$200/月的订阅计划在促销期间可产生 $13-15k 的 API 等效用量，但超出限额后会被切断，不建议购买额外用量。
- **反对/质疑：**
  - 许多评论者质疑“24/7 运行 AI agent 的实际用途”，认为缺乏 killer app。常见用例（如扫描日志、自动提 PR）被认为过于小众或可以用更简单的方式完成。
  - 有评论认为，给 agent sudo 权限是鲁莽的，建议使用 unprivileged 账户或 VLAN 隔离。
  - 有评论指出，对于不需要 macOS 特定功能的场景，使用 $3 VPS 或 libvirt VM 更简单、更易重置。
- **风险/限制：** 即使物理隔离，agent 仍可访问本地网络。建议将其置于独立 VLAN 或 deny-all 防火墙规则之后。

### What AI did to stackoverflow in a graph

352 pts · 408 comments · [data.stackexchange.com](data.stackexchange.com)

**📌 内容总结**

- 背景：通过 Stack Exchange Data Explorer 查询 Stack Overflow 每月新增问题数量，生成时间序列图。
- 关键要点：问题数量在 2014 年达到峰值（约 207K/月），此后持续下降。2022 年底 ChatGPT 发布后，下降速度显著加快。最新数据点已降至约 588/月。
- 实际结论：AI 加速了 SO 的衰落，但并非唯一原因。SO 的下降趋势早在 2016-2017 年就已开始。

**💬 讨论总结**

- **共识：SO 的衰落是多重因素叠加的结果，AI 是“最后一根稻草”。** 主要因素包括：
  - **社区文化问题：** 过度严格的审核、对新用户不友好、问题被随意关闭或标记为重复。许多用户表示“被 SO 的审核员劝退”。
  - **知识饱和：** 随着问题空间被填满，新问题数量自然下降。SO 的目标是成为知识库，而非社区。
  - **替代方案兴起：** Reddit、GitHub Issues、Discord 等渠道分流了问答需求。
  - **AI 的冲击：** ChatGPT 等 LLM 提供了即时、无需等待的答案，进一步削弱了 SO 的吸引力。
- **历史背景：** SO 在 2021 年被 Prosus 以 18 亿美元收购。收购后，SO 的管理层做出了不受欢迎的 AI 相关决策（如禁止 AI 生成内容、后来自行推出 AI 功能），进一步疏远了核心用户。
- **反对/质疑：**
  - 有评论认为，SO 的下降曲线在 2022 年底出现明显加速，AI 是直接原因。
  - 有评论指出，SO 的 SEO 在 2020 年左右被 Google 算法更新破坏，导致流量下降。
  - 有评论认为，SO 的衰落是“好事”，因为其审核文化已经病态，AI 只是加速了其必然的消亡。
- **商业现实：** SO 通过向 AI 公司出售数据（如与 Google 的交易）来变现其积累的知识库，但这并未能挽救其社区活力。
