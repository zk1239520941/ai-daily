### 1️⃣ 腾讯混元300天变革：97后姚顺雨空降，重构组织发布Hy3

* **核心发布**：晚点LatePost发布深度报道，详述97年出生的姚顺雨空降腾讯混元300天的改革历程。姚顺雨此前为OpenAI研究员，提出ReAct和思维树，2025年底加入腾讯担任大语言模型负责人。
* **改革措施**：三个月内更换预训练、后训练、评估、Infra全部负责人；成立专门服务混元的AI Infra部和AI Data部；提出"守正出奇"训练思路，先回归常识做对基础事。混元团队层级简化为"姚顺雨—方向负责人—研究员/实习生"三层。
* **成果与挑战**：Hy3 Preview上线后登顶OpenRouter周度使用榜，单周处理3.66万亿Token。正式版Hy3在小参数规模基础上赶超GLM5.1。Hy4已在训练中，尺寸更大且加入多模态能力，预计2027年进入国内第一梯队。
* **组织变革**：AI Lab被并入混元，重新定义为Frontier前沿研究团队，不设KPI。姚顺雨提出模型与产品联合设计（Co-design）模式，向元宝等业务派驻后训练团队。

### 2️⃣ Hugging Face Transformers模型现可在vLLM中以原生速度运行

* **核心发布**：Hugging Face CEO Clement Delangue宣布，Transformers模型现可在vLLM中以原生速度运行，性能匹配或超越手写实现。此前每种新架构需在Transformers和vLLM中分别实现。
* **性能数据**：在4B到235B参数的模型上，Transformers后端匹配或超越原生vLLM吞吐量，包括张量并行和MoE设置。一个可读的模型实现即可同时支持训练、微调、评估、RL和推理。
* **行业意义**：这消除了模型开发到生产部署之间的重复实现成本，使模型作者只需在Transformers中实现一次架构即可立即获得vLLM的优化推理栈。

### 3️⃣ 商汤开源SenseNova-Vision：视觉任务大统一，超越Vision Banana

* **核心发布**：商汤科技正式发布并全面开源SenseNova-Vision理解生成统一视觉大模型，首次将检测、分割、深度预测、3D重建等经典视觉任务原生融入多模态底座。
* **技术路线**：不同于行业此前将多个专家模型打包封装的"统一视觉"，SenseNova-Vision让视觉成为通用基础模型的原生能力。所有经典视觉任务统一表述为多模态生成问题，无需为不同任务设计专属预测头。
* **性能表现**：在目标检测、指代检测、OCR、关键点定位等任务上全面领先同类型通用模型。深度估计、表面法向估计精度达到几何专用模型水准。在稠密小目标检测、长尾类别识别等复杂场景下表现尤为突出。
* **开源生态**：同步开源包含5000万条高质量样本的视觉指令语料库SenseNova-Vision Corpus-50M。模型和数据均全面开源。
 🔗 [魔搭ModelScope社区报道](https://mp.weixin.qq.com/s?__biz=Mzk3NTc1NTU0Mw==&mid=2247510661&idx=1&sn=ab8e4817b7d8adceed914cd36c17ed74) | [GitHub](https://github.com/OpenSenseNova/SenseNova-Vision)

### 4️⃣ 腾讯混元开源HyOCR-1.5：1B参数端到端OCR，推理提速6.37倍

* **核心发布**：腾讯混元正式开源HyOCR-1.5端到端OCR大模型，仅1B参数覆盖8种以上text-centric任务，推理最高提速6.37倍，在OmniDocBench v1.6以94.74分稳居端到端第一。
* **技术亮点**：引入基于DFlash的投机解码框架，90.7M参数的轻量级block-diffusion草稿模型一次并行前向即可"猜"出一整块候选token。在Transformers下取得6.37×加速，vLLM下取得2.14×加速。
* **能力扩展**：通过Agentic Data Flow（智能体驱动的数据流）系统扩展能力边界，覆盖低资源语种OCR（331种语言）、古文字OCR（汉字七体之变）、多图问答等长尾场景。
* **部署灵活性**：支持通过llama.cpp在CPU、消费级显卡乃至普通笔记本上运行，实现"随处可部署"。

基于 tree-sitter 实现 40+ 语言的确定性 AST 解析，零 LLM 成本完成代码图构建；采用 Leiden 社区检测算法自动划分子系统。

### Apple's new SpeechAnalyzer API, benchmarked against Whisper and its predecessor

412 pts · 172 comments · [get-inscribe.com](https://get-inscribe.com/blog/apple-speech-api-benchmark.html)

**📌 内容总结**

- 作者运营一款同时集成 Apple 新旧语音引擎和 WhisperKit 的转录应用，借此在相同生产代码路径下，对五个引擎进行了 LibriSpeech 基准测试（5,559 条语料，全设备端运行于 M2 Pro）。
- Apple 未公布 SpeechAnalyzer 的精度数据，迁移决策此前依赖猜测。测试结果：新 API 在干净语音上 WER 2.12%（旧 API 9.02%），噪声语音 4.56%（旧 API 16.25%），同时以约 3 倍于 Whisper Small 的速度运行并击败后者。
- 作者公开了所有原始逐句转录结果，Whisper 列与 OpenAI 公布数据偏差在 +0.11 到 +0.42 之间，验证了测试方法的可靠性。局限性：仅限英语、朗读语音、单机测试。

**💬 讨论总结**

- **共识观点**：Apple 新 API 在英语语音识别上确实显著优于旧 API，且对大多数用户而言，迁移是明确的。
- **工程经验**：
  - 旧 API 默认将音频发送至 Apple 服务器，需手动设置 `requiresOnDeviceRecognition` 才能强制设备端运行。
  - 新 API 使用后必须调用 `finalizeAndFinishThroughEndOfInput()` 才能结束会话，否则会永久挂起——作者在自己的产品中发现了这个 bug。
- **历史背景**：Whisper Small/Base/Tiny 已发布近四年，未随 Whisper v2/v3 更新，并非当前最优模型。多位评论者指出应对比 Parakeet、Nemotron、Voxtral、Cohere Transcribe 等更近期的模型。
- **反对意见**：
  - 仅对比 Whisper 小模型（Small/Base/Tiny）而不包括 Large V3 或 V3 Turbo，测试不完整。Whisper Large V3 Turbo 可在旧款 MacBook 上实时运行且精度更高。
  - 多语言支持是 Whisper 的核心优势（100+ 语言 vs. SpeechAnalyzer 约 30 个区域），且 Whisper 可跨平台运行。
  - 有评论者认为 Parakeet TDT 0.6b v2/v3 在精度和速度上已优于 Whisper，且可在旧款 iPhone 15 Pro Max 上良好运行。
- **商业现实**：对开发者而言，使用 Apple 内置模型的最大优势是不需将模型打包进应用，节省磁盘空间；用户只需下载一次，所有应用共享。

  - 作者将 Nobie 与 Excel 的关系类比为 V8 与 JavaScript：Nobie 是运行时，而 openpyxl 只是文件格式库，无法维护公式依赖、图表等语义。
