### GLM5.2 on AMD MI355X at 2626 tok/s/node at over 2x lower cost than Blackwell

39 pts · 11 comments · [wafer.ai](https://www.wafer.ai/blog/glm52-amd)

**📌 内容总结**

- Wafer 是一家 GPU 云服务商，长期推广 AMD MI355X 作为 NVIDIA B300/B200 的低成本替代方案。本文展示了他们在 AMD 平台上优化 GLM-5.2 推理的具体工程实践和性能数据。
- HN 关注点:
  - AMD MI355X 在特定工作负载下达到 B200 约 80% 的吞吐量，但成本低 2 倍以上（单 GPU 价格约为 B300 的 1/2.75）。
  - 优化过程涉及 MXFP4 量化（AMD Quark）、推理引擎选型（sglang vs vLLM vs ATOM）、投机解码（speculative decode）修复、MoE kernel 调优等工程细节。
  - 关键数据：单流 213 tok/s（GLM-5.2, 10k in/1.5k out），聚合吞吐 2626 tok/s/node @ 2.4 RPS（20k in/1k out, 60% cache hit）。

**💬 讨论总结**

- **共识观点**：AMD 在性价比上有明显优势，但软件生态（ROCm、框架支持）仍是主要短板。CUDA 护城河正在被侵蚀，但尚未被填平。
- **工程经验**：
  - 优化过程并非“开箱即用”：需要手动修复 sglang ROCm 镜像中的投机解码 bug（MTP head 量化前缀不匹配、CUDA header 硬编码），以及手动调优 MoE kernel 选择。
  - 本次优化未编写自定义 kernel（与之前 Qwen3.5 397B 的工作不同），说明 AMD 软件栈在快速进步。
- **商业现实**：
  - Wafer 员工透露其毛利率约 40%，利用率是决定利润率的关键因素。
  - 有评论指出，AMD 在非美国市场（电力成本高、NVIDIA 供应受限）可能有独特机会，但 AMD GPU 的采购渠道和软件可靠性仍是未知数。
- **反对/质疑**：
  - 有用户指出，AMD 消费级硬件的软件支持仍然糟糕（llama.cpp 的 Vulkan 后端比 ROCm 后端快 5-10%），对企业级硬件持保留态度。
  - 2600 tok/s 是聚合吞吐量，单用户实际体验为 213 tok/s。

### Jamesob's guide to running SOTA LLMs locally

257 pts · 121 comments · [github.com/jamesob](https://github.com/jamesob/local-llm)

**📌 内容总结**

- 作者想做什么：分享自己搭建本地 SOTA LLM 推理机器的完整硬件配置、软件调优和工程经验。
- 解决的问题：在本地运行接近 Claude Opus 水平的开源模型（GLM-5.2-594B），避免依赖闭源 API。
- 技术实现：
  - **$2k 方案**：2× RTX 3090（48GB VRAM），可运行 Qwen3.6-27B。
  - **$40k 方案**：4× RTX PRO 6000（384GB VRAM），运行 REAP 剪枝 + NVFP4 量化的 GLM-5.2（~594B 参数），~80 tok/s @ 460k context。
  - 关键硬件：c-payne PCIe Gen4 交换机实现 GPU 间 P2P 直连（27.5 GB/s 单向，50.4 GB/s 双向，0.37-0.45 µs 延迟），避免数据经过 CPU root complex。
  - 大量 BIOS/内核调优：禁用 IOMMU、ACS、ASPM，强制 PCIe Gen4 链路速度，GPU 功率限制（350W/卡）以在 110V 电路上运行。
- 用户反馈焦点：社区对“$40k 方案是否值得”争议极大，核心矛盾在于量化/剪枝后的模型质量损失 vs 本地可控性。

**💬 讨论总结**

- **共识观点**：
  - 本地 LLM 的价值在于隐私、无限制使用、无 API 成本焦虑，但当前性价比远不如云服务。
  - 量化（4-bit）和剪枝（REAP）会带来可感知的质量损失，尤其在长上下文编码任务中，小错误会累积。
- **工程经验**：
  - PCIe 交换机配置是最大难点：需要 BIOS 设置、内核参数、运行时 ACS 禁用、SAS 线缆选型等多重调优。
  - 4× RTX 6000 实际功耗约 1400W，需注意电路容量。
  - 有用户分享 Intel Arc B70（32GB, $1k）可跑 Qwen3.6 35B Q4 达 88 tok/s，是低成本替代方案。
- **历史背景**：
  - 有评论将当前本地 AI 比作“Altair 和 IMSAI 时代”——硬件已可用，但缺乏 Apple II 级别的开箱即用体验。
  - 多位用户提到 Apple M 系列芯片（M4/M5 Max）是“中间方案”：128GB 统一内存可跑 DeepSeek V4 flash，但内存带宽仅为 GPU 的 1/3 到 2/3。
- **商业现实**：
  - $40k 硬件成本相当于 16.8 年 Claude Opus $200/月订阅，但若按企业 API 定价（~$4k/月）则仅 10 个月。
  - 多位用户指出，租用云 GPU 比自建更经济、灵活，且无需处理硬件维护。
- **反对/质疑**：
  - 多位用户质疑“SOTA”定义：GLM-5.2 的 REAP 版本已剪掉约 22% 的专家，且量化到 4-bit，与原始 BF16 模型性能差距显著。
  - 有用户指出，真正运行完整 GLM-5.2（1.5TB VRAM）需要 8× H200（~$400k+），$40k 方案只能跑“被阉割的版本”。
  - 有评论认为，在量化模型上运行长上下文编码任务时，质量下降“令人痛苦”。
  - 部分用户认为，对于大多数任务（摘要、简单脚本），小模型（Gemma 4 12B Q4）已足够，无需追求 SOTA。

## 今日洞察

**地缘政治正在重塑AI工具链的默认选择。** 阿里巴巴全面禁用Claude Code，直接原因是安全研究员发现的Unicode水印机制——这并非孤立事件。腾讯内部AI额度从2000美金降至2000元，中国公司正系统性地从闭源海外模型转向国内及开源方案。这不是"技术选型"，而是供应链重组：当工具链底层存在不可控的识别机制，开发者行为会被迫迁移。Claude Code在编码代理市场的高增长可能在中国市场被硬性截断，而替代品（Kimi K2.7 Code刚接入GitHub Copilot）的窗口正在打开。

**Anthropic启动自研芯片，标志着AI实验室的算力博弈进入新阶段。** 谷歌有TPU，亚马逊有Trainium，微软有Maia，Meta也在自研——Anthropic是最后一个还在完全依赖别人芯片的主要实验室。挖来OpenAI定制芯片团队早期成员、与三星讨论2nm工艺，说明其目标不是通用加速器，而是针对推理工作负载的专用芯片。这背后的约束很直接：推理成本正在成为Agent产品规模化的瓶颈，自研芯片是唯一能从根本上压低边际成本的路径。但芯片设计周期通常3-5年，Anthropic在短期内的算力谈判筹码仍取决于其采购规模。

**Meta出售AI算力的信号比表面更复杂。** 市场将其解读为"AI算力稀缺叙事动摇"，但更准确的解读是：Meta的1250-1450亿CapEx需要对冲。如果内部AI应用变现不及预期，过剩算力可以变成云收入；如果内部需求旺盛，云业务可以随时收缩。这是一张期权，而非方向性转变。真正值得关注的是：当Meta这样的超大规模玩家开始考虑"算力过剩"场景，意味着行业对推理需求的增长预期可能已经触顶——至少短期内，供给增速超过了需求增速。
